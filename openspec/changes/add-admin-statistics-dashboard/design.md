## Context

当前 `src/pages/Dashboard/index.jsx` 仅为静态欢迎文案，无法承载管理运营视角的数据洞察。项目使用 React + Ant Design + MobX，用户基础信息来自 `store.user.userBaseInfo`（包含 `userRoles`、`schoolId`），已具备按角色控制展示逻辑的基础。后端已提供平台与学校两个统计域接口，前端需聚合 9 个接口并输出可视化结果。

该变更是典型的跨模块改造：涉及页面重构、HTTP API 封装扩展、角色逻辑、图表库接入与状态编排，且需要明确平台角色与学校角色的展示边界。

## Goals / Non-Goals

**Goals:**
- 在管理端仪表盘提供可运营的统计分析视图，覆盖平台视角与学校视角。
- 基于登录角色控制视角可见性：学校管理员仅学校视角，平台管理员/超管支持 Tab 切换且默认平台视角。
- 接入 `@ant-design/plots` 输出关键图表与核心指标卡，确保信息结构清晰、可快速决策。
- 统一处理加载态、错误态与空态，避免多接口页面出现“局部失真”。

**Non-Goals:**
- 不扩展教师视角或学生视角统计页面。
- 不改动后端统计接口和统计口径。
- 不实现实时推送、导出报表等高级分析能力。

## Decisions

### 1. 页面分层：容器页 + 视角子面板
- 决策：`Dashboard` 作为容器层，仅负责角色识别、Tab 状态与筛选条件；平台视角和学校视角拆分为独立子组件（例如 `PlatformOverviewPanel`、`SchoolOverviewPanel`）。
- 原因：单组件承载 9 个接口和多类图表会导致维护成本与回归风险显著上升。
- 备选方案：单文件集中实现。未采纳原因：状态耦合严重，后续扩展教师/学生视角时成本过高。

### 2. 角色判定策略：以 `userRoles` 为单一事实来源
- 决策：从 `store.user.userBaseInfo.userRoles` 识别角色。
  - 平台角色：`root`、`admin`
  - 学校角色：`school_root`、`school_admin`
- 展示策略：
  - 平台角色显示双 Tab，默认 `platform`
  - 学校角色仅展示 `school`
- 备选方案：通过菜单权限推导角色。未采纳原因：菜单权限不等同于页面视角权限，且会引入隐式耦合。

### 3. 数据请求编排：按视角分组并行 + 局部缓存
- 决策：
  - 平台视角进入时并行请求 5 个平台接口。
  - 学校视角进入时并行请求 4 个学校接口。
  - Tab 首次切换触发对应视角加载，已加载视角在筛选条件不变时复用数据。
- 原因：减少首屏阻塞，避免一次性请求 9 个接口造成无效网络开销。
- 备选方案：首屏全量请求。未采纳原因：学校管理员并不需要平台数据，浪费带宽与渲染时间。

### 4. API 封装策略：在 `src/http/api.ts` 增加显式方法
- 决策：新增 9 个统计 API 函数，统一使用 `http.get(path, { params })`，并在类型文件中补充 DTO 类型。
- 原因：保持与现有 API 风格一致，减少页面中散落字符串路径。
- 备选方案：页面内直接写 `http.get`。未采纳原因：不利于复用与测试。

### 5. 图表映射策略：指标卡 + 轻量图表组合
- 决策：使用 `@ant-design/plots` 进行以下可视化映射：
  - 学校入驻分布空心饼图（申请/通过/驳回）并与合作学校总量卡片同排
  - 存储占比（视频 vs 普通文件）
  - 平台用户规模（总数/教师/学生对比）
  - 学院分布（教师数/学生数分组柱图）
  - 比率指标（上架率、提交率、得分率）
- 原因：比纯数字更易呈现趋势与结构。
- 备选方案：仅 Statistic 卡片。未采纳原因：结构信息表达不足。

### 6. 查询参数策略：统一可选时间范围，学校视角可选 schoolId
- 决策：支持 `startTime`、`endTime` 透传到所有统计接口；学校视角支持可选 `schoolId` 参数（平台角色可指定，学校角色默认不传）。
- 原因：与接口协议一致，满足后续按周期分析。
- 备选方案：首期不做时间筛选。未采纳原因：会限制运营分析价值，且接口已支持。

### 7. 数值标准化策略：统一百分比与容量显示
- 决策：
  - 比率字段统一展示为百分比（保留两位小数）。
  - 存储统一转为易读单位（GB/TB）并保留原始字节值用于 tooltip。
- 原因：后端返回值单位不一时可避免展示误解。
- 备选方案：原样显示。未采纳原因：可读性差且易误判。

## Risks / Trade-offs

- [比率字段口径不一致（0~1 或 0~100）] → 在前端增加口径归一化函数，并为异常值输出告警日志与兜底展示。
- [平台角色学校视角缺少 schoolId 输入来源] → 首版提供可选输入框（手工输入学校 ID），后续可接学校列表远程搜索。
- [多接口并行导致部分成功部分失败] → 采用“局部容错 + 总览提示”，失败卡片显示重试入口，不阻塞其他卡片。
- [图表在小屏场景可读性下降] → 配置自适应高度、横向滚动容器与简化标签策略。

## Migration Plan

1. 在 `src/http/api.ts` 与类型定义中新增统计接口方法与 DTO。
2. 重构 `src/pages/Dashboard/index.jsx`，拆分视角子组件与公共指标卡组件。
3. 接入 `@ant-design/plots` 图表，完成平台/学校两套数据映射。
4. 增加筛选条件（时间范围、可选 schoolId）并联动数据刷新。
5. 按角色进行回归验证：`root`、`admin`、`school_root`、`school_admin`。
6. 发布后观察接口失败率与页面渲染耗时，必要时降级为仅指标卡显示。

回滚策略：若出现线上阻断问题，可回退 Dashboard 页面至静态欢迎版本（保留 API 封装不启用），不影响其他模块。

## Clarifications

- 平台角色在学校视角 MUST 先输入并确认 `schoolId` 后才允许加载学校数据。
- 时间筛选默认范围采用“最近 30 天”，并提供“最近 7 天 / 最近 30 天”快捷选项。
- 学院分布数据为空时，隐藏学院分布图表区域。
- 所有比率类字段统一保留 2 位小数展示。
