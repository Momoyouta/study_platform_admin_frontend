## Context

当前后台仅有分片上传链路，缺少针对 `file_chunk` 的统一治理入口。此次变更同时涉及：
- 前端：新增一级菜单、路由、列表页、筛选表单、操作弹窗、角色差异化交互。
- 后端（接口契约层面）：新增查询/重命名/删除/迁移接口，约束学校管理员与平台管理员的数据范围。
- 文件存储：迁移操作不仅更新数据库 `target_path`，还要移动真实文件，且路径规则受 `type` 影响。

约束条件：
- 维持现有 React + Antd + Axios 的页面与请求组织方式。
- 管理端接口统一走 `/admin/fileChunk/*`。
- 学校管理员与平台管理员共享同一页面，但可见/可用能力不同。

## Goals / Non-Goals

**Goals:**
- 提供“文件管理”单层菜单入口与对应列表页。
- 支持按指定条件分页查询并支持三种排序字段。
- 支持单项重命名（仅 `fileName`）、删除（含物理文件）、迁移到学校资源库（仅 `status=done`）。
- 在同一接口协议下完成角色数据范围隔离：学校管理员仅可操作所属学校；平台管理员必须先提供学校 ID。
- 列表返回并展示 `creatorName`、`schoolName`，不展示分片序号相关字段。

**Non-Goals:**
- 不修改分片上传初始化、分片上传、合并流程。
- 不新增批量删除、批量迁移、文件预览下载等增强能力。
- 不调整课程资源绑定业务，仅处理文件治理和归档迁移。

## Decisions

### 1) 页面结构采用“单页列表 + 行级操作”模式
- 决策：新增文件管理页面，沿用现有后台 list 页范式（筛选区 + 表格 + 分页 + 行操作）。
- 原因：与既有页面交互一致，学习成本低，改造范围可控。
- 备选方案：做成“查询页 + 详情页”双页流。
- 放弃原因：会引入额外路由与状态同步成本，且本需求以运维型快速操作为主。

### 2) 查询权限在前后端双重约束
- 决策：
  - 前端：学校管理员隐藏/禁用学校 ID 输入并自动注入所属学校 ID；平台管理员未输入学校 ID 时禁止检索。
  - 后端：二次校验角色与 schoolId 关系，防止绕过前端。
- 原因：单端校验不可靠，双重约束可避免越权。
- 备选方案：仅后端校验。
- 放弃原因：虽然安全，但交互反馈滞后，平台管理员误操作成本高。

### 3) 迁移路径策略按文件类型分流并保持 hash 离散化
- 决策：
  - `type=1` 迁移至 `resource_library/videos`。
  - `type=2` 迁移至 `resource_library/documents`，同样采用 hash 命名 + 二级目录离散。
- 原因：与既有视频资源库规则一致，可降低热点目录压力并保持运维可预期。
- 备选方案：documents 使用原始文件名平铺存储。
- 放弃原因：冲突风险高、目录膨胀快，且与视频规则不一致。

### 4) 删除能力采用“记录删除 + 物理文件删除”原子意图，提供 force 兜底
- 决策：默认删除时必须尝试删除真实文件；若传 `force=true`，允许在物理文件异常时继续清理记录。
- 原因：兼顾数据一致性与故障场景下的可恢复操作。
- 备选方案：仅删除数据库记录。
- 放弃原因：会遗留孤儿文件，长期增加存储成本与排障复杂度。

### 5) 接口与类型设计遵循“字段最小暴露”
- 决策：列表返回项不包含分片数与已上传分片序号集合，仅保留管理所需字段并补充 `creatorName`、`schoolName`。
- 原因：降低 payload 与前端复杂度，避免无用字段误用。
- 备选方案：返回完整 `file_chunk` 结构。
- 放弃原因：传输冗余且增加前端耦合。

## Risks / Trade-offs

- [风险] 文件迁移涉及磁盘移动，跨盘或权限不足可能失败。
  - 缓解：迁移前校验源路径存在性和目标目录可写；失败时返回明确错误码并保持记录不变。
- [风险] 学校管理员身份下自动注入 schoolId 依赖登录态字段完整性。
  - 缓解：后端以 token 中的实际归属学校为准，不信任前端入参。
- [权衡] 平台管理员强制输入 schoolId 才可检索，牺牲部分便利性。
  - 缓解：增加表单提示文案和必填校验，防止空查询拉全量数据。
- [风险] 删除接口在 `force=true` 下可能掩盖文件系统异常。
  - 缓解：记录审计日志，标记为强制删除并附带异常摘要。

## Migration Plan

1. 新增后端接口与 DTO，先在测试环境完成联调。
2. 新增前端菜单、路由、页面与 API 封装，灰度给平台管理员。
3. 验证学校管理员权限边界（查询、编辑、删除、迁移均应受限于所属学校）。
4. 执行回归：现有上传与课程资源流程不受影响。
5. 上线策略：先发布后端，再发布前端；回滚时先回滚前端，后端保留兼容窗口。

## Open Questions

- 迁移接口返回的 `targetPath`/`filePath` 是否统一为相对根目录路径（当前按接口草案处理）。
- `file_chunk` 的 `type` 为空时按已确认策略处理：允许查询展示，但 MUST 禁止迁移操作。
- 平台管理员的 `schoolId` 查询输入组件按已确认策略保留可扩展点，后续允许平滑替换为“学校下拉选择器”以降低输入错误率。
