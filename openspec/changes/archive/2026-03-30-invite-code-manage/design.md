## Context

当前邀请码管理页面已支持按类型创建邀请码，但“学生加入课程”场景缺少课程维度字段，导致创建请求无法稳定关联课程。现有创建弹窗中字段展示与校验规则为静态配置，无法对特定类型施加动态必填约束。

## Goals / Non-Goals

**Goals:**
- 为邀请码实体与创建请求补充 `course_id` 字段。
- 当邀请码类型为“学生加入课程”时，动态展示课程ID输入项并强制必填。
- 保持其他邀请码类型的创建流程不受影响。

**Non-Goals:**
- 不改造邀请码列表查询、分页、删除等已有能力。
- 不引入课程下拉选择、课程名称回填或远程课程合法性校验。
- 不调整现有权限矩阵与角色过滤逻辑。

## Decisions

### 决策 1：以邀请码类型驱动课程ID字段显示与校验
- 方案：在创建表单中监听 `type` 值；当 `type === 2`（学生加入课程）时渲染 `course_id` 表单项，并附加 `required` 规则。
- 原因：与当前 Ant Design Form 模式兼容，交互直观，改动集中在页面组件。
- 备选方案：始终展示 `course_id` 并在提交时手动校验。未采用原因是会增加无关类型的认知负担，且容易遗漏即时错误提示。

### 决策 2：提交前进行字段归一化，避免脏数据透传
- 方案：构建 payload 时按 `type` 分支处理 `course_id`：课程邀请码必须携带，其他类型清空或不传。
- 原因：避免用户先选择课程类型后切换为其他类型时残留 `course_id`，保证接口语义一致。
- 备选方案：完全依赖后端忽略无效字段。未采用原因是会导致前后端契约不清晰，增加排错成本。

### 决策 3：类型定义与接口契约同步扩展
- 方案：若项目中存在邀请码请求/实体类型定义（如 `src/type/invite.ts`），同步增加 `course_id` 可选或条件必填描述。
- 原因：保持业务字段在 UI、类型与 API 语义上的一致性，减少后续维护歧义。
- 备选方案：仅在页面内使用匿名对象。未采用原因是可读性与可维护性较差。

## Risks / Trade-offs

- [风险] 前端类型枚举与后端类型常量不一致导致条件判断失效
  - Mitigation：复用既有 `InviteTypeMap` 枚举来源，并在提交逻辑中使用明确数值比较。
- [风险] 动态字段切换后出现旧值残留，误提交 `course_id`
  - Mitigation：在类型切换时重置 `course_id`，并在 payload 构建时二次过滤。
- [风险] 只做前端必填仍可能被绕过
  - Mitigation：在设计中明确后端需对课程邀请码执行 `course_id` 必填校验（前后端双保险）。

## Migration Plan

1. 更新前端创建表单：新增动态 `course_id` 字段显示与必填规则。
2. 更新创建 payload 组装逻辑：在课程邀请码场景透传 `course_id`。
3. 补充/更新类型定义（如存在）。
4. 执行手工回归：三种邀请码类型创建、字段切换、缺失字段拦截。
5. 发布后观察创建失败日志与参数上报，必要时快速回滚到旧表单逻辑。

- 已确认：`course_id` 按字符串类型处理，前端不做数值化约束。
