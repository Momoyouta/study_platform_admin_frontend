# 规范：邀请码 API

## GHERKIN 风格验收标准
### 场景：创建邀请码
**给定** 管理员在邀请码管理页面
**当** 他们填写类型、学校 ID 和 TTL
**并且** 点击“创建”
**那么** 一个新的邀请码被生成并添加到列表中。

### 场景：搜索邀请码
**给定** 管理员输入搜索条件（如邀请码、学校 ID）
**当** 他们点击“搜索”
**那么** 列表仅显示符合条件的邀请码。

### 场景：删除邀请码
**给定** 邀请码列表
**当** 管理员点击特定邀请码上的“删除”
**并且** 确认删除
**那么** 邀请码将从系统中移除。

## 详细要求
### 要求：基于角色的可见性
- 平台管理员 (root, admin) 必须看到并填写 `school_id` 字段。
- 学校管理员 (school_root, school_admin) 不得看到 `school_id` 字段；它应自动填充自个人资料。

### 要求：过期逻辑
- 如果 `当前时间 - 创建时间 > ttl`，邀请码即为“无效”。
- UI 应根据此计算结果显示“有效”或“无效”标签。

### 要求：TTL 转换
- 创建弹窗必须允许用户选择“天”或“小时”。
- 在发送到 API 之前，该值必须转换为秒。
