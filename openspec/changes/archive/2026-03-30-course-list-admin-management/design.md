## Context

当前后台已有多类列表页（用户、学校、邀请码等）的通用交互范式，但课程管理下的课程列表能力尚未落地。业务要求本次先交付可运营的最小闭环：可分页查询、可创建课程、可切换发布状态；编辑流程先占位，后续再扩展到课程详情页并联动章节/课时/作业等多表数据。

受约束点：
- 查询条件必须克制，仅支持课程名关键词、状态、school_id。
- 平台管理员/平台超级管理员查询前必须输入 school_id，避免跨校全量扫描。
- 接口风格需保持现有项目约定（统一响应结构、页面侧 .then() 链式调用习惯）。

## Goals / Non-Goals

**Goals:**
- 在课程管理页建立标准化课程列表能力，支持分页条件查询。
- 提供创建课程入口与创建弹窗，完成最小字段录入与提交。
- 在列表操作栏支持课程状态切换（未发布/已发布）。
- 保留“编辑”操作入口并明确后续跳转详情页的扩展方向。

**Non-Goals:**
- 不实现课程详情编辑页。
- 不实现章节、课时、作业等关联实体编辑。
- 不新增任课老师、章节数、课时数筛选能力。
- 不纳入硬删除与关联清理流程。

## Decisions

### 决策 1：沿用“搜索区 + 表格区”列表骨架并引入角色门禁校验
- 方案：复用现有列表页模式，搜索区提供 keyword、status、school_id；在点击查询前，对平台管理员/平台超级管理员执行 school_id 必填校验，未通过则提示并阻断请求。
- 原因：与现有页面保持一致，学习成本最低，同时满足平台角色必须指定学校的业务约束。
- 备选方案：由后端兜底返回缺参错误。未采用原因：用户体验较差，且会产生无效请求。

### 决策 2：创建课程采用页内弹窗而非独立路由页
- 方案：在课程列表页提供“创建课程”按钮，使用 Modal + Form 提交 createCourseAdmin。
- 原因：本期创建字段少（name 必填，其余可选），页内弹窗可降低路由与状态管理复杂度，便于快速上线。
- 备选方案：跳转独立创建页。未采用原因：实现成本更高，不利于本次最小闭环交付。

### 决策 3：状态切换统一走 updateCourseAdmin，编辑操作仅做占位
- 方案：操作列提供“发布/下架”切换，提交 updateCourseAdmin({ id, status })；“编辑”按钮保留但本期不进入实际编辑流程，仅提示“建设中”或禁用点击。
- 原因：与现有接口契约一致，能在不引入复杂编辑逻辑的情况下满足运营侧上下架需求。
- 备选方案：使用 softDeleteCourseAdmin 作为下架语义。未采用原因：语义偏“删除”，与页面“状态切换”行为不完全一致。

### 决策 4：接口与类型先最小增量补齐，为后续详情编辑留扩展点
- 方案：补齐 listCourseAdmin/createCourseAdmin/updateCourseAdmin 的 API 封装与课程类型定义，列表数据结构兼容 teacher_names/chapter_count/total_lesson_count 等展示字段，但查询表单不暴露对应筛选。
- 原因：既满足当前列表展示需要，也避免后续接入课程详情时重复改造类型层。
- 备选方案：仅使用页面内匿名对象。未采用原因：可维护性差，后续扩展风险高。

## Risks / Trade-offs

- [风险] 平台角色未输入 school_id 导致查询逻辑分叉复杂
  - Mitigation：封装统一的查询参数构造函数，在发起请求前集中校验与拦截。
- [风险] 创建课程时 school_id 规则与后端权限判断不一致
  - Mitigation：前端按角色给出显式提示，并保留后端错误信息透传。
- [风险] “编辑”入口占位可能引发用户误解
  - Mitigation：按钮文案/提示明确“暂未开放，后续跳转详情页编辑”。
- [风险] 状态切换与软删除语义存在边界重叠
  - Mitigation：本期只暴露发布状态切换，不在界面中混入删除操作。

## Migration Plan

1. 新增/补齐课程管理 API 方法与课程类型定义。
2. 在课程列表页实现查询表单、表格分页、状态切换逻辑。
3. 接入创建课程弹窗与创建接口提交，成功后刷新列表。
4. 增加平台角色 school_id 必填校验与提示文案。
5. 联调并完成回归：角色差异查询、分页、创建、状态切换、编辑占位交互。
6. 若上线后出现阻塞问题，可临时关闭“创建课程”入口并保留只读列表能力。
